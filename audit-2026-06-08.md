## 🔴 CRITICAL — Дыры безопасности (немедленное исправление)

### CRIT-1: SSH root-пароль в открытом виде (множественные места)

**Где найдено:**
- `memory/MEMORY.md`: `user: root, pass: 19735` — впрыскивается в контекст LLM каждый старт сессии
- `memory-maintenance` cron (фаза 0): пароль 13 раз вшит в payload джобы через `sshpass -p '19735'`
- `memory-maintenance` cron (фаза 1): ещё 7 вхождений

**Риск:** Любой, кто получит доступ к контексту LLM (логи, memory dump, shared chat), получит root-доступ ко всей инфраструктуре.

**Исправление:** SSH-ключи вместо пароля. Пароль убрать из всех payload'ов и MEMORY.md.

### CRIT-2: Пароль CouchDB в открытом виде в кроне

**Где:** `memory-maintenance` фаза 1.4:
```
curl -s http://judy:gN5sR0tX2qM8kV9p@127.0.0.1:5984/judy
```
Реальный пароль CouchDB вшит в payload cron-джобы → виден в `jobs.json`, в контексте LLM при выполнении.

**Исправление:** Убрать пароль из payload. Использовать `docker exec judy-couchdb curl ...` или передавать через env.

### CRIT-3: MCP-сервер открыт миру (0.0.0.0:8765)

**Что:** `judy-mcp` слушает на `0.0.0.0:8765`, без аутентификации, без TLS. Любой в локальной сети (и интернете если нет фаервола) может:
- Читать/писать/удалять память (memory_store, memory_recall, memory_delete)
- Читать/писать Obsidian заметки
- Управлять инвентарём
- Читать/писать Redis

**Исправление:** `127.0.0.1:8765` + SSH-туннель или mTLS для внешнего доступа.

### CRIT-4: Админский пароль Postgres в init.sql

**Где:** `services/postgres/init.sql`:
```sql
CREATE ROLE admin WITH SUPERUSER ... PASSWORD 'Admin1234';
```
**Риск:** Хардкод-пароль в репозитории. Если кто-то получит доступ к PostgreSQL (а порт хоть и на 127.0.0.1, но внутри LXC доступен), получит полный контроль над БД.

**Исправление:** Убрать admin-роль из init.sql или генерировать пароль из env.

### CRIT-5: dbgate БЕЗ аутентификации на 0.0.0.0:5052

**Что:** Полноценный GUI-клиент для БД, доступен всем в сети без пароля. Можно подключиться к PostgreSQL, выполнить любой SQL.

**Исправление:** Добавить `DBGATE_AUTH=password` + переменные окружения, или закрыть на 127.0.0.1.

### CRIT-6: Ключи API в openclaw.json

**Где:**
- `env.DEEPSEEK_API_KEY: sk-622e...` 
- `gateway.auth.token: a8594cd4...`
- `plugins.brave.config.webSearch.apiKey: BSAGZj...`

**Риск:** openclaw.json — не секретный файл. Он лежит в репозитории, читается плагинами, отдаётся в debug-логах. Утечка = угон API-ключа DeepSeek + полный перехват gateway + расход чужих Brave-токенов.

**Исправление:** Вынести все секреты в `.env`/секрет-менеджер, в openclaw.json оставить только `${VAR}` ссылки.

### CRIT-7: Grafana с анонимным доступом на 0.0.0.0:3000

**Что:** `GF_AUTH_ANONYMOUS_ENABLED: true` + `GF_AUTH_ANONYMOUS_ORG_ROLE: Viewer`. Любой в сети видит дашборды без логина. Даже Viewer видит сессионные токены, имена каналов, метрики инфраструктуры.

**Исправление:** Выключить анонимный доступ или закрыть порт на 127.0.0.1.

### CRIT-8: CouchDB CORS: '*' (open bar)

**Что:** `cors/origins: "*"` + `cors/credentials: true`. Любой сайт может делать CORS-запросы к CouchDB с кредами. Классическая CSRF-уязвимость.

**Исправление:** Ограничить origins до конкретных доменов (openclaw.home.maxweek.ru).

## 🟠 HIGH — Надёжность и архитектура

### HIGH-1: Единая точка отказа — весь стек в одном LXC

**Ситуация:** 10 контейнеров (postgres, couchdb, redis, mcp, embeddings, prometheus, grafana, dbgate, app-api, app-web) в одном LXC на 6GB/64GB. 

**Что сломается:** Падение LXC = потеря всего сразу. OOM на одном сервисе = каскадное падение остальных. 6GB RAM для pgvector + CouchDB + Redis + TEI-модель + Prometheus — на грани.

**Рекомендация:** Вынести критичное (postgres, embeddings) в отдельные LXC. Минимум — увеличить RAM до 12GB.

### HIGH-2: Нет бэкапов — катастрофа при отказе диска

**Что проверено:** Ни одного backup-скрипта, ни одного cron'а на pg_dump, ни volume-снапшотов в docker-compose. 

- Postgres: 25+ записей, 8MB — потеряем всю долгосрочную память
- CouchDB: ~240 документов структурированных заметок — потеряем все Obsidian-заметки
- Redis: in-memory, потеря при рестарте приемлема

**Рекомендация:** 
1. `pg_dump` ежедневно в `/backups/postgres/` с ротацией 30 дней
2. CouchDB backup через `couchdb-dump` или volume snapshot
3. SCP на отдельную машину или rclone в облако
4. Добавить в docker-compose backup-контейнер с cron

### HIGH-3: Postgres — проблема стабильности пароля

**История:** За последнюю неделю — минимум 2 инцидента с `password authentication failed for user "judy"`. Root cause: рассинхрон между CI/CD vars и реальным паролем в контейнере.

**Почему:** В CI/CD используется `vars.ENV` (GitHub CI/CD variables), но пересоздание контейнера postgres с тем же volume сохраняет старый пароль, а новый (из CI/CD) не применяется, потому что `POSTGRES_PASSWORD` читается только при первом создании БД.

**Исправление:** 
1. Добавить в deploy скрипт проверку: после деплоя — `docker exec judy-postgres psql -U judy -c "SELECT 1"`, если fail → пересоздать с флагом
2. Перейти на фиксированный пароль, который НЕ меняется между деплоями

### HIGH-4: MCP server — соединения к Postgres без пула

**Что:** `server.py` создаёт новое подключение на каждый вызов: `conn = await asyncpg.connect(PG_DSN)` → `await conn.close()`. При нескольких одновременных вызовах — множественные коннекты, overhead на TLS/аутентификацию.

**Рекомендация:** `asyncpg.create_pool(min_size=2, max_size=10)` в глобальной переменной.

### HIGH-5: Redis maxmemory 512mb с allkeys-lru — тихая потеря данных

**Что:** `--maxmemory 512mb --maxmemory-policy allkeys-lru`. Если Redis заполнится, он молча выкинет least-recently-used ключи. Нет алерта, нет лога.

**Рекомендация:** Добавить мониторинг used_memory/maxmemory ratio в Prometheus (redis_exporter). Настроить алерт на >80%.

### HIGH-6: Embeddings — silent failure без алертов

**Что:** `_get_embedding()` возвращает `None` при ошибке, и memory_store сохраняет запись БЕЗ эмбеддинга (только `[no embedding]` в ответе). При этом факт остаётся в Postgres, но не будет найден векторным поиском никогда.

**Рекомендация:** 
1. Метрика: embedding_failures_total 
2. Алерт при высоком failure rate
3. Healthcheck embeddings через Prometheus

## 🟡 MEDIUM — Точки роста

### MED-1: Только векторный поиск — нет multi-signal retrieval

**Что:** Сейчас поиск в Postgres: cosine similarity по pgvector. То есть «смысловой» поиск. Если запрос не совпадает по смыслу с эмбеддингом — результат пустой (fallback на полнотекстовый ts_vector и ILIKE).

**Индустрия (Mem0, 2026):** Multi-signal retrieval = semantic similarity + BM25 keyword + entity matching, fused score.

**Рекомендация:** Добавить keyword/entity matching к текущему пайплайну memory_recall. Извлекать сущности при записи и индексировать отдельно.

### MED-2: Нет memory scoping / multi-tenancy

**Что:** Все памяти в одном постгресе, без разделения на user/agent/session/org. Для одного Макса — ок. Для Home Assistant + Telegram + ещё агентов — будет каша.

**Индустрия:** Mem0 4-scope model (user_id, agent_id, run_id, app_id). Langfuse: project-level isolation.

**Рекомендация:** Добавить поле `scope` в таблицу memories и фильтровать при recall. Минимум — `agent_id`.

### MED-3: Нет memory staleness detection

**Что:** Факты в Postgres не имеют «срока годности». Запись двухмесячной давности и сегодняшняя равны для векторного поиска.

**Индустрия:** temporal scoring (более новые записи имеют boost), staleness flags, contradiction resolution.

**Рекомендация:** Добавить decay factor в scoring: `importance * exp(-days_since_creation / 30)`.

### MED-4: Нет мониторинга качества памяти

**Что:** Мы не знаем:
- Какой recall rate у memory_recall? (сколько раз нашли vs не нашли)
- Сколько дубликатов?
- Какая средняя latency эмбеддинга?

**Рекомендация:** Добавить счётчики: recall_hits, recall_misses, embedding_latency_ms, duplicate_count. Вывести на Grafana.

### MED-5: memory_flush просит LLM, а не автоматизирован

**Что:** `memoryFlush.enabled: true` срабатывает при компакшене контекста, но это prompt → LLM решает, сохранять или нет. Нет гарантии сохранения.

**Рекомендация:** Автоматическое сохранение ключевых паттернов без участия LLM (хеш последних N фактов → если новый → сохранить).

### MED-6: CouchDB chunked storage — можно потерять часть заметки

**Что:** `obsidian_write` разбивает контент на чанки по 200 символов, каждый чанк — отдельный документ в CouchDB. Потеря одного чанка = битая заметка.

**Рекомендация:** Добавить целостность: хранить хеш полного содержимого в родительском документе, верифицировать при `obsidian_read`.

### MED-7: Нет traceability для LLM-вызовов

**Что:** Не записывается:
- Какая модель использовалась
- Сколько токенов потрачено
- Какие тулзы были вызваны
- Latency каждого вызова

**Индустрия:** Langfuse, MLflow, OpenObserve — LLM observability с трассировкой.

**Рекомендация:** Добавить Langfuse самому в docker-compose (open source, 21K★). Прост как 2 копейки: контейнер + конфиг.

## 🟢 LOW — Качество жизни

### LOW-1: memory/MEMORY.md содержит критичные секреты

Уже охвачено CRIT-1, но шире: MEMORY.md — это индексный файл, который впрыскивается в контекст каждый раз. Он НЕ должен содержать секретов. Сейчас содержит SSH-пароль.

### LOW-2: CouchDB healthcheck использует curl без аутентификации

`curl -sf http://localhost:5984/` — работает даже без пароля? Стоит проверять аутентифицированный запрос.

### LOW-3: TEI (text-embeddings-inference) использует CPU-образ

`cpu-1.5` — эмбеддинги на CPU. Модель `multilingual-e5-base` — 278M параметров. Если Макс добавит GPU в Proxmox, можно перейти на `cuda-1.5` → x10 ускорение.

### LOW-4: Docker images без фиксации версий

`image: redis:7-alpine` — может обновиться до минорной версии при перезапуске. `pgvector/pgvector:pg17` — аналогично. Без фиксации хеша есть риск breaking change при деплое.

## 📈 Масштабирование (Home Assistant, Telegram, другие агенты)

### Что нужно для multi-agent:

1. **Memory scoping (CRITICAL):** Добавить `agent_id`/`source` в memories. Агент Home Assistant не должен видеть личные заметки Макса из Obsidian.

2. **Rate limiting на MCP:** Сейчас любой агент может спамить memory_store без ограничений. Нужен rate limiter (token bucket).

3. **Connection pool:** При 3+ агентах, одновременно обращающихся к Postgres, без пула соединений база ляжет.

4. **Выделенный embeddings сервис:** CPU-модель на одном ядре не выдержит 3+ параллельных запросов на эмбеддинг.

5. **Redis namespace isolation:** `judy:{key}` → `judy:homeassistant:{key}`, `judy:telegram:{key}` и т.д. Уже частично сделано.

6. **Obsidian multi-writer:** CouchDB CouchDB-движок поддерживает multi-master репликацию, но текущий `obsidian_write` не атомарный (чанки пишутся без транзакции). Нужна очередь или локи.

## 🔍 Мониторинг — что добавить

### Уже есть:
- ✅ Prometheus (скрапит openclaw-exporter)
- ✅ Grafana (один дашборд — сессионные токены)
- ✅ Docker healthchecks на всех контейнерах
- ✅ Ночной maintenance-отчёт (ручной)

### Чего нет:

1. **AlertManager** — Prometheus есть, алертов нет. Настроить на:
   - Диск > 80%
   - Память > 85%  
   - Контейнер down > 60s
   - Postgres connections > 80%
   - Embedding failure rate > 5%

2. **Postgres exporter** — метрики БД (connections, locks, query latency, index usage, dead tuples)

3. **Redis exporter** — used_memory, hit rate, evicted keys

4. **cAdvisor / node_exporter** — метрики LXC (CPU, RAM, disk I/O, network) — критично для мониторинга OOM

5. **Embeddings latency** — p50/p95/p99 времени ответа TEI

6. **MCP health dashboard** — success/failure rate по каждому memory тулу

7. **LLM cost tracking** — сколько токенов/денег уходит на auto-capture + maintenance + обычные запросы

8. **Disk growth prediction** — при текущем темпе (21% за ~2 недели) — когда закончится диск?

## 💎 Итог

**Система:** Умная, многослойная (4 слоя памяти: файлы + pgvector + Obsidian + Redis), с продуманной ночной консолидацией. Архитектурно грамотная — лучше чем у 80% self-hosted проектов.

**Проблема:** Безопасность на уровне «прототип, которым пользуется один человек». 8 критических дыр делают систему уязвимой при малейшем расширении (второй агент, внешний доступ, новый пользователь).

**Вердикт:** Не готова к production-grade использованию. Но с исправлением CRIT-1..8 и добавлением бэкапов становится уверенной домашней системой. С точками роста MED-1..7 — системой уровня малого бизнеса.

**Немедленное действие №1:** Убрать пароли из текстовых файлов и контекста LLM. Всё остальное может подождать день-два.
